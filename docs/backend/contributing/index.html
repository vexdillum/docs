Contributing

Документ фиксирует правила командной разработки проекта Vexd.

Ветки​

В проекте используется упрощенный GitLab Flow.

  • dev - основная ветка разработки.
  • main - стабильная ветка для защиты и деплоя.
  • Рабочие ветки создаются от dev.
  • Изменения попадают в dev через merge request.
  • Перенос стабильной версии из dev в main выполняется отдельным merge request.

Названия рабочих веток должны показывать смысл задачи:

  • feature/<name> - новая функциональность;
  • fix/<name> - исправление ошибки;
  • docs/<name> - документация;
  • infra/<name> - инфраструктура и деплой;
  • допускаются короткие имена зон работ, например admin_panel или scoring-v2.

Issues​

Задачи ведутся в GitLab Issues.

Для задачи указывается:

  • краткое описание;
  • исполнитель;
  • метка;
  • срок или milestone, если он применим.

Ветка должна логически соответствовать issue или отдельной зоне работ.

Коммиты​

Сообщение коммита должно коротко объяснять, что изменено.

Желательный формат:

area: short action

Примеры:

auth: add login validation
leaderboard: fix cache invalidation
admin: add user ban endpoint

Один коммит должен относиться к одной логической задаче или небольшому связанному набору изменений.

Merge Request​

Перед merge request нужно проверить:

  • что ветка создана от актуального dev;
  • что изменения относятся к заявленной задаче;
  • что не добавлены секреты, локальные файлы и временные артефакты;
  • что CI проходит без ошибок.

Если изменение затрагивает API, нужно обновить Swagger/OpenAPI.

Проверки​

Перед merge желательно выполнить:

gofmt -w .
go vet ./...
golangci-lint run ./... --timeout 5m
go test ./...

Coverage считается по основным backend-пакетам командой, указанной в README.md.

Секреты и окружение​

В репозиторий нельзя добавлять реальные секреты:

  • .env;
  • токены;
  • пароли;
  • приватные kubeconfig-файлы;
  • production-ключи.

Для примеров используются только .env.example и другие шаблоны без реальных значений.

Сгенерированные файлы​

Сгенерированные файлы коммитятся только если они нужны для запуска, проверки или демонстрации проекта.

Перед коммитом нужно убедиться, что в diff нет случайных временных файлов, логов, локальных отчетов и IDE-настроек.
