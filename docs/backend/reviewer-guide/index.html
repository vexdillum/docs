Проверочные команды​

Локальный запуск:

Copy-Item .env.example .env
docker compose up -d --build
docker compose ps
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8080/ping

Swagger/OpenAPI:

Invoke-Item docs\swagger.html
go run ./cmd/docs

Тесты и статический анализ:

go test ./...
go vet ./...
golangci-lint run ./... --timeout 5m

Coverage считается по основным backend-пакетам командой из testing.md. Такой набор исключает main, router wiring, generated code и генератор документации.

Логи контейнеров:

docker compose logs --tail=120 api flagchecker
docker compose exec -T api sh -c "tail -n 80 /tmp/lost-last/api.log"
docker compose exec -T flagchecker sh -c "tail -n 80 /tmp/lost-last/flagchecker.log"

Реализованные backend-контуры​

  • Регистрация и вход пользователей.
  • Создание, изменение, удаление команды и управление участниками.
  • Список заданий, получение задания по id, фильтрация и кеширование каталога.
  • Submit флага через отдельный FlagChecker service.
  • Защита от повторного solve после смены команды.
  • First blood и динамический scoring на уровне solve/scoring данных.
  • Leaderboard с кешированием и последними solve.
  • Team dashboard с кешированием.
  • Admin CRUD заданий.
  • Admin ban/unban/hard delete пользователя.
  • Contest start/end/freeze/unfreeze.
  • Admin управление dynamic challenge lifecycle через kCTF boundary.
  • Admin управление активными инстансами.
  • Audit log действий администратора.
  • Prometheus metrics и Grafana dashboard.

Важные объяснения​

Почему FlagChecker отдельно​

Submit флагов является пиковой нагрузкой во время Flag Rush. Если проверка флага, запись solve и scoring будут жить только внутри основного HTTP API, то API станет единственной точкой давления. Отдельный gRPC-сервис позволяет масштабировать проверку флагов отдельно и держать HTTP handler проще.

Почему scoring идет через очередь​

Полный пересчет очков и инвалидирование нескольких кешей не должны блокировать пользовательский submit. Поэтому после solve создается событие, а worker пересчитывает score асинхронно.

Почему leaderboard и dashboard кешируются​

Эти страницы часто обновляются участниками. Готовое представление в Redis дешевле, чем каждый раз собирать данные из нескольких таблиц PostgreSQL.

Почему есть два вида логов​

Файловые structured logs нужны для технической диагностики сервиса. Audit log в PostgreSQL нужен для бизнес-событий админки и отображения в интерфейсе.
