Архитектура backend

Проект реализует backend для соревнований Jeopardy CTF. Основная идея: участники работают с HTTP API, проверка флагов вынесена в отдельный gRPC-сервис, а дорогие операции scoring/cache invalidation выполняются асинхронно через outbox и очередь.

Компоненты runtime​

КомпонентПутьРоль
APImain.goHTTP API на Hertz для участников и администраторов
FlagCheckercmd/flagcheckergRPC-сервис проверки флагов и записи solves
PostgreSQLexternal serviceОсновное хранилище пользователей, команд, заданий, solves, audit
Redisexternal serviceКэш leaderboard, catalog, team dashboard
RabbitMQexternal serviceОчередь событий для scoring/outbox
kCTF adapterinternal/kctfГраница интеграции с Kubernetes/kCTF

Основной поток отправки флага​

sequenceDiagram
    participant U as Participant
    participant API as Hertz API
    participant FC as FlagChecker gRPC
    participant DB as PostgreSQL
    participant O as Outbox
    participant MQ as RabbitMQ
    participant W as Scoring worker
    participant R as Redis

    U->>API: POST /catalog/challenges/:id/submit
    API->>FC: SubmitFlag(user_id, challenge_id, flag)
    FC->>DB: transaction: validate user/team/challenge/flag, record solve
    FC->>O: create scoring event
    FC-->>API: status correct/wrong
    API->>R: invalidate team/catalog/leaderboard caches
    O->>MQ: publish scoring event
    MQ->>W: consume scoring event
    W->>DB: recalculate challenge/team scores
    W->>R: invalidate leaderboard/team dashboard/catalog caches

Почему flagchecker вынесен отдельно:

  • API остается тонким HTTP-контуром.
  • Проверка флага и запись solve изолированы.
  • На пиках нагрузки flagchecker можно масштабировать отдельно от панели.
  • gRPC дает строгий контракт между API и сервисом проверки.

Scoring и outbox​

Scoring не пересчитывается полностью в HTTP handler. После записи solve создается событие, которое обрабатывается worker-ом.

Ключевые модули:

  • internal/service/flagchecker - проверка флага и запись solve.
  • internal/service/outbox - публикация событий из outbox.
  • internal/service/scoring - обработка scoring jobs.
  • internal/repo/postgres_scoring_repo.go - пересчет score в PostgreSQL.
  • internal/messaging/rabbitmq - RabbitMQ publisher/consumer.

Такой подход уменьшает время ответа submit endpoint и отделяет пользовательский запрос от тяжелого пересчета leaderboard.

Кэширование​

Redis используется для read-heavy частей:

КэшПутьЧто хранит
Leaderboardinternal/cache/boardcacheГотовые данные leaderboard
Challenge cataloginternal/cache/challengecacheВидимые пользователю задания
Team dashboardinternal/cache/teamcacheDashboard команды

Кэш инвалидируется после solve, изменения scoring-параметров, изменения команды и административных действий, влияющих на отображение.

Contest lifecycle​

Contest settings хранятся в PostgreSQL и управляют доступностью действий:

  • регистрация;
  • просмотр каталога;
  • submit флагов;
  • публичность scoreboard;
  • freeze/unfreeze leaderboard.

Ключевые модули:

  • internal/service/contest
  • internal/repo/contest_repo.go
  • biz/handler/ctf/contest_handlers.go
  • admin endpoints в biz/handler/admin.go

Административный контур​

Админка закрыта middleware-цепочкой:

  1. Auth middleware проверяет cookie/JWT.
  2. Admin middleware проверяет роль пользователя.
  3. CSRF middleware защищает mutating endpoints.

Админские возможности:

  • обзор состояния системы;
  • управление contest lifecycle;
  • CRUD заданий;
  • deploy/stop/status/access/logs для dynamic challenges;
  • ban/unban/hard delete пользователей;
  • просмотр команд, solves, активных инстансов;
  • управление инстансами;
  • просмотр audit log.

Audit и логирование​

Есть два разных типа логов.

Технические логи:

  • пишутся в stdout и файл;
  • содержат timestamp, level, service, message;
  • API пишет access-log по каждому HTTP-запросу;
  • настраиваются через LOG_FILE, LOG_LEVEL, LOG_DEBUG.

Бизнес-аудит админки:

  • хранится в таблице admin_audit_logs;
  • содержит actor_user_id, actor_username, actor_email, action, target_type, target_id, payload, created_at;
  • доступен через GET /api/v1/admin/audit.

Технический лог отвечает на вопрос "что происходило в процессе", audit отвечает на вопрос "какой администратор какое бизнес-действие совершил".

kCTF boundary​

Backend не должен напрямую смешивать HTTP handler и Kubernetes API. Для этого есть интерфейс kctf.Client.

Он поддерживает:

  • Deploy
  • Stop
  • Access
  • Logs

Если kubeconfig недоступен, используется stub-client. Это позволяет тестировать admin API без реального Kubernetes-кластера.

Основные проектные решения​

РешениеПричина
HertzНеблокирующий сетевой стек и высокая производительность на HTTP I/O
FlagChecker как отдельный gRPC-сервисИзоляция проверки флагов и возможность масштабирования отдельно от API
Outbox + RabbitMQНадежная доставка событий scoring без выполнения тяжелой работы в HTTP handler
Redis-кэш готовых представленийСнижение нагрузки на PostgreSQL на read-heavy страницах
Admin audit в БДАдминские действия должны быть доступны для интерфейса и не зависеть от ротации файловых логов
kCTF через интерфейсВозможность заменить stub на Kubernetes-клиент без изменения handlers
