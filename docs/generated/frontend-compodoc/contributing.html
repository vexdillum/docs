Contributing

This project uses GitLab Flow.

Branches

  • main: stable release branch.
  • dev: integration branch for tested features.
  • feature/<issue-number>-<short-name>: new user-facing feature.
  • fix/<issue-number>-<short-name>: bug fix.
  • test/<issue-number>-<short-name>: test-only work.
  • docs/<issue-number>-<short-name>: documentation-only work.
  • chore/<issue-number>-<short-name>: tooling, config or cleanup.

Examples:

Example :
feature/24-rating-page
fix/31-auth-token-refresh
test/42-team-page-coverage
docs/45-deploy-guide

Each active participant should have at least one assigned GitLab Issue and at least one working branch during the project.

Issues

Every GitLab Issue should contain:

  • a clear title;
  • a short description of the expected result;
  • acceptance criteria;
  • assignee;
  • labels;
  • due date;
  • links to related merge requests.

Close an issue only after the related merge request is merged.

Commits

Use small commits. One commit should describe one logical task.

Commit message format:

Example :
#<issue-number> <imperative action>

Examples:

Example :
#24 add rating page tests
#31 fix auth token header
#45 document GitLab Pages deployment

Avoid messages such as upd, fix, changes, final or hotfix without context.

Merge Requests

Merge requests go into dev first. After integration testing, dev is merged into main for release.

Before requesting review:

Example :
npm run lint
npm run test:coverage
npm run build

The merge request description should include:

  • linked issue;
  • summary of changes;
  • testing results;
  • screenshots for visible UI changes;
  • deployment notes if the change affects build or runtime configuration.

Review Rules

Reviewers check:

  • whether the implementation matches the issue;
  • whether tests cover the changed behavior;
  • whether the UI still works in the browser;
  • whether the branch has a clear name;
  • whether commit messages reference issues.

Do not merge if linting, tests or production build fail.

Release

Release flow:

Example :
feature/fix branch -> merge request -> dev -> CI checks -> main -> GitLab Pages

After a release, keep the GitLab Pages deployment available for at least 48 hours and keep CI logs for technical defense evidence.

Ru

Участие в разработке

Проект использует GitLab Flow.

Ветки

  • main: стабильная release-ветка.
  • dev: интеграционная ветка для проверенных фич.
  • feature/<issue-number>-<short-name>: новая пользовательская функциональность.
  • fix/<issue-number>-<short-name>: исправление ошибки.
  • test/<issue-number>-<short-name>: изменения только в тестах.
  • docs/<issue-number>-<short-name>: изменения только в документации.
  • chore/<issue-number>-<short-name>: инструменты, конфигурация или cleanup.

Примеры:

Example :
feature/24-rating-page
fix/31-auth-token-refresh
test/42-team-page-coverage
docs/45-deploy-guide

У каждого активного участника должна быть минимум одна назначенная GitLab Issue и минимум одна рабочая ветка за время проекта.

Issues

Каждая GitLab Issue должна содержать:

  • понятный заголовок;
  • краткое описание ожидаемого результата;
  • критерии приёмки;
  • исполнителя;
  • labels;
  • due date;
  • ссылки на связанные merge requests.

Закрывайте issue только после merge связанного merge request.

Коммиты

Делайте небольшие коммиты. Один коммит должен описывать одну логическую задачу.

Формат сообщения коммита:

Example :
#<issue-number> <imperative action>

Примеры:

Example :
#24 add rating page tests
#31 fix auth token header
#45 document GitLab Pages deployment

Избегайте сообщений вроде upd, fix, changes, final или hotfix без контекста.

Merge Requests

Merge requests сначала вливаются в dev. После интеграционного тестирования dev вливается в main для релиза.

Перед запросом review выполните:

Example :
npm run lint
npm run test:coverage
npm run build

Описание merge request должно включать:

  • связанную issue;
  • краткое описание изменений;
  • результаты тестирования;
  • скриншоты для видимых UI-изменений;
  • заметки по деплою, если изменение влияет на сборку или runtime-конфигурацию.

Правила review

Проверяющие смотрят:

  • соответствует ли реализация issue;
  • покрывают ли тесты изменённое поведение;
  • работает ли UI в браузере;
  • понятно ли названа ветка;
  • ссылаются ли сообщения коммитов на issues.

Не вливайте merge request, если linting, тесты или production-сборка падают.

Релиз

Порядок релиза:

Example :
feature/fix branch -> merge request -> dev -> CI checks -> main -> GitLab Pages

После релиза держите GitLab Pages deployment доступным минимум 48 часов и сохраните CI-логи как доказательство для технической защиты.
