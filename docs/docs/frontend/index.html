Lost Last Frontend​

Frontend for a CTF platform. The application shows challenges, teams, users, team pages, user profiles, rating tables and authentication pages.

Stack​

  • Angular 21
  • TypeScript 5.9
  • Carbon Design System
  • Tailwind CSS
  • Transloco for localization
  • ECharts for charts
  • Vitest and Angular test runner
  • ESLint and angular-eslint

Requirements​

Use Node.js 24 or newer and npm 10 or newer. The recommended runtime is fixed in .nvmrc, .node-version, package.json and GitLab CI.

nvm use
npm ci

If nvm is not installed, install Node.js 24+ manually and then run npm ci.

Development​

Start the local frontend:

npm start

By default Angular starts the app at http://localhost:4200.

The frontend API base URL is configured in src/shared/config/api.config.ts. For local backend proxying, use proxy.conf.json if the backend is started on a separate port.

Scripts​

npm run lint
npm test
npm run test:coverage
npm run activity:check
npm run docs
npm run build
npm run build:development
  • lint checks TypeScript and Angular templates.
  • test runs unit tests.
  • test:coverage runs unit tests with coverage.
  • activity:check verifies Git activity for technical defense.
  • docs generates Compodoc HTML documentation in documentation/ from Angular component metadata, including inputs and outputs.
  • build creates a production build in dist/CTFast/browser.
  • build:development creates a development build.

Quality Gates​

The project is expected to pass these checks before merge:

npm run lint
npm run test:coverage
npm run build

Current local verification on 2026-05-17:

  • lint: passed
  • unit tests: 123 passed
  • coverage: 92.8% statements, 86.28% branches, 91.35% functions, 92.13% lines
  • production build: passed
  • Compodoc docs: generated in documentation/
  • npm audit: 0 vulnerabilities

Angular persistent build cache is disabled in angular.json. This keeps the build reproducible in CI and avoids native cache crashes observed on macOS with the current Angular builder.

Project Structure​

src/app/              Angular app bootstrap, routes and providers
src/pages/            Route-level pages
src/widgets/          Reusable UI widgets
src/entities/         Shared TypeScript interfaces
src/utils/            API clients, guards and interceptors
src/shared/config/    Runtime configuration
src/assets/i18n/      Localization dictionaries
src/test-*.ts         Shared test setup
public/               Static assets copied to the build

CI/CD​

GitLab CI is configured in .gitlab-ci.yml.

Pipeline stages:

  1. activity: verifies repository activity for technical defense.
  2. quality: runs ESLint.
  3. test: runs ng test --coverage --watch=false and publishes coverage/ artifacts.
  4. build: creates the production frontend bundle.
  5. docs: generates Compodoc HTML documentation in documentation/.

Pipeline artifacts:

  • coverage/
  • dist/CTFast/browser/
  • documentation/

Pipeline Scheme​

GitLab commit -> GitLab CI -> npm ci -> activity check -> lint -> тесты с покрытием -> production build -> Compodoc docs

The pipeline is reproducible because all dependencies are installed from package-lock.json, Node.js is fixed in CI, and both the static build output and generated documentation are produced directly by CI.

Activity Evidence​

The activity_check CI job runs npm run activity:check. It checks Git history for at least four non-holiday active weeks with at least five commits per week, which matches the repository activity threshold for a two-person frontend team.

Git Workflow​

The team uses GitLab Flow with dev as the integration branch and main as the release branch. Detailed branch, commit and merge request rules are described in CONTRIBUTING.md.

Lost Last Frontend​

Фронтенд для CTF-платформы. Приложение показывает задания, команды, пользователей, страницы команд, профили пользователей, рейтинговые таблицы и страницы авторизации.

Стек​

  • Angular 21
  • TypeScript 5.9
  • Carbon Design System
  • Tailwind CSS
  • Transloco для локализации
  • ECharts для графиков
  • Vitest и Angular test runner
  • ESLint и angular-eslint

Требования​

Используйте Node.js 24 или новее и npm 10 или новее. Рекомендуемая версия окружения зафиксирована в .nvmrc, .node-version, package.json и GitLab CI.

nvm use
npm ci

Если nvm не установлен, установите Node.js 24+ вручную и затем выполните npm ci.

Разработка​

Запуск локального фронтенда:

npm start

По умолчанию Angular запускает приложение на http://localhost:4200.

Базовый URL API настраивается в src/shared/config/api.config.ts. Для локального проксирования backend используйте proxy.conf.json, если backend запущен на отдельном порту.

Скрипты​

npm run lint
npm test
npm run test:coverage
npm run activity:check
npm run docs
npm run build
npm run build:development
  • lint проверяет TypeScript и Angular-шаблоны.
  • test запускает unit-тесты.
  • test:coverage запускает unit-тесты с покрытием.
  • activity:check проверяет Git-активность для технической защиты.
  • docs генерирует Compodoc HTML-документацию в documentation/ из метаданных Angular-компонентов, включая inputs и outputs.
  • build создаёт production-сборку в dist/CTFast/browser.
  • build:development создаёт development-сборку.

CI/CD​

GitLab CI настроен в .gitlab-ci.yml.

Этапы pipeline:

  1. activity: проверка активности репозитория для технической защиты.
  2. quality: запуск ESLint.
  3. test: запуск ng test --coverage --watch=false и публикация artifacts из coverage/.
  4. build: production-сборка фронтенда.
  5. docs: генерация Compodoc HTML-документации в documentation/.

Артефакты pipeline:

  • coverage/
  • dist/CTFast/browser/
  • documentation/

Схема pipeline​

GitLab commit -> GitLab CI -> npm ci -> activity check -> lint -> tests with coverage -> production build -> Compodoc docs

Pipeline воспроизводим: зависимости устанавливаются из package-lock.json, версия Node.js зафиксирована в CI, а статическая сборка и документация генерируются самим pipeline.

Подтверждение активности​

CI job activity_check запускает npm run activity:check. Проверка смотрит историю Git и требует минимум четыре активные недели вне майских праздников с минимум пятью коммитами в неделю, что соответствует порогу активности для фронтенд-команды из двух человек.

Git Workflow​

Команда использует GitLab Flow: dev как интеграционная ветка и main как release-ветка. Подробные правила веток, коммитов и merge requests описаны в CONTRIBUTING.md.
